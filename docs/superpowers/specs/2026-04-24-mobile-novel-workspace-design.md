# 手机端小说创作工作区支持设计

## 背景

当前应用的整体布局以桌面端为主：顶部栏固定高度、左侧项目导航或小说工作区导航固定宽度，主内容区使用桌面滚动容器。小说编辑页和章节编辑页承载完整创作链路，但在手机端会出现导航挤压、三栏编辑器无法阅读、按钮和工具入口过密等问题。

本次第一阶段选择“创作工作区优先”：重点优化小说编辑页 `/novels/:id/edit` 和章节编辑页 `/novels/:id/chapters/:chapterId`。其他页面暂不做移动端专属体验，避免扩大范围。

## 目标

- 手机端可以顺畅进入小说创作工作区，切换项目设定、故事宏观规划、角色准备、卷骨架、节奏拆章、章节执行、质量修复和版本历史。
- 手机端章节编辑页优先保证正文阅读、编辑、保存和 AI 修改入口可用。
- 移动端实现尽量独立存放，避免把大量响应式规则散落进桌面组件。
- 桌面端现有布局、导航和默认行为保持不变。
- 业务状态、AI 调用、任务状态、章节数据逻辑继续复用现有容器和 hooks，不复制核心业务流程。

## 非目标

- 不为全站所有管理页建立移动端专属体验。
- 不重写小说工作区业务逻辑、AI 调用逻辑、任务执行逻辑或数据获取逻辑。
- 不调整桌面端 `Sidebar`、`NovelWorkspaceRail`、`Navbar` 的默认展示和交互。
- 不在第一阶段做移动端完整视觉重设计或新信息架构。

## 技术方案

采用“移动端专属壳 + 复用现有内容”的方案。

### 布局分流

在应用布局入口新增一个轻量的屏宽判断组件，例如 `MobileResponsiveGate`。它只负责判断当前是否处于手机断点：

- 桌面端继续渲染现有 `Navbar`、`Sidebar` / `NovelWorkspaceRail` 和桌面 `main` 容器。
- 手机端仅在小说工作区路由启用移动端壳。
- 非小说工作区路由在第一阶段仍走现有布局，避免引入全站范围变更。

断点判断作为 UI 展示判断，不参与业务路由、任务调度或 AI 决策。

### 文件边界

新增移动端目录，保持实现集中：

- `client/src/components/layout/mobile/*`
  - `MobileResponsiveGate.tsx`
  - `MobileWorkspaceShell.tsx`
  - `MobileWorkspaceHeader.tsx`
  - `MobileWorkspaceBottomBar.tsx`
- `client/src/pages/novels/mobile/*`
  - `MobileNovelStepNav.tsx`
  - `MobileNovelEditView.tsx`
  - `MobileChapterEditorShell.tsx`
  - 需要时增加移动端工具抽屉或分段控件

桌面组件只允许做必要的入口连接，例如在 `NovelEdit` 中按移动端状态选择 `NovelEditView` 或 `MobileNovelEditView`。不把移动端布局 class 大量加入桌面组件内部。

## 小说编辑页设计

`MobileNovelEditView` 接收与 `NovelEditView` 相同的 props，复用现有 Tab 组件和操作回调。

页面结构：

1. 顶部区域显示小说标题、当前步骤、AI 推荐步骤和任务状态提醒。
2. `MobileNovelStepNav` 使用 `NOVEL_WORKSPACE_FLOW_STEPS` 和版本历史工具项，横向滚动展示创作步骤。
3. 内容区域渲染当前 active tab 对应的现有业务面板。
4. 工具入口将导出、项目工具、任务抽屉、版本历史等低频能力集中到移动端工具菜单。

交互原则：

- 默认突出当前步骤和 AI 推荐步骤，帮助新手知道下一步做什么。
- 当前页面主要行动按钮保留在可见区域，低频操作进入工具入口。
- 现有 Tab 的业务组件如 `BasicInfoTab`、`StoryMacroPlanTab`、`ChapterManagementTab`、`PipelineTab` 暂不复制逻辑；第一阶段只通过移动端外壳降低导航和页面容器挤压。

## 章节编辑页设计

`MobileChapterEditorShell` 接收与 `ChapterEditorShell` 相同的 props，复用保存、AI 修改、诊断定位、候选应用等核心逻辑。

桌面端现有 `ChapterEditorShell` 是三栏布局：左侧章节信息和诊断，中间正文编辑器，右侧 AI 导演面板。手机端改为分段视图：

- “正文”：默认展示，包含章节正文编辑器、保存状态和保存入口。
- “AI 建议”：展示推荐任务、修正指令、候选版本和应用入口。
- “诊断”：展示质量诊断卡、定位问题片段、运行诊断和版本历史入口。

交互原则：

- 手机端默认进入“正文”，保证写作和阅读面积最大。
- 选择正文片段后，AI 修改入口应清晰可见，但不遮挡正文。
- 保存状态和未保存提醒需要始终可感知。
- AI 生成中、应用候选中、保存中等状态沿用现有状态提示，不新增业务状态。

## 数据与状态流

- `NovelEdit` 继续作为小说工作区状态容器，负责数据查询、mutation、SSE、工作流同步和 props 组装。
- `NovelChapterEdit` 继续负责章节编辑所需数据查询和路由回退。
- 移动端组件只消费现有 props 和回调，不直接新增业务 API 调用。
- 移动端路由仍使用现有 URL，不增加平行移动端 URL。

## 错误处理

- 加载、失败、空数据状态沿用现有页面状态。
- 手机端工具菜单或抽屉打开失败时，不影响主内容区域。
- 断点判断不可用时默认走桌面渲染，避免空白页。
- 移动端新增组件不吞掉现有 toast、任务抽屉和 AI 流程反馈。

## 测试与验证

第一阶段验证重点：

- 类型检查通过。
- 桌面宽度下，小说编辑页和章节编辑页仍渲染现有桌面组件。
- 手机宽度下，小说编辑页显示移动顶部、横向步骤导航和当前 Tab 内容。
- 手机宽度下，章节编辑页默认显示正文编辑区，能保存正文，能进入 AI 建议和诊断区域。
- 桌面与手机断点切换时不丢失当前 active tab、章节选择和正文草稿。
- 新增 UI 文案从用户视角描述动作和下一步，不出现改造、迁移、已升级等开发过程表达。

## 实施顺序

1. 增加移动端断点判断和移动端布局目录。
2. 接入小说工作区路由的移动端分流，确保桌面仍走原路径。
3. 实现移动端小说编辑页外壳和步骤导航。
4. 实现移动端章节编辑页分段结构。
5. 补充必要的类型或组件级测试，并运行类型检查。
6. 回归桌面小说编辑页和章节编辑页，确认未受影响。
